## Architecture Pattern

**Feature-Slice Design (FSD)** — code is organized by feature (`src/features/`), with each feature containing `api/`, `model/`, and `ui/` sub-layers. Shared utilities live in `src/shared/`.

## Components

### Auth (`src/features/auth/`)
- **Purpose:** Passwordless OTP login. Requests OTP via email (enqueued as a `Job`), verifies code, sets `session_id` cookie.
- **Status:** Partially implemented. OTP verification is hardcoded to `'123456'` — real DB check is a TODO. OTP is never actually stored for verification; only enqueued to be emailed.
- **Session:** Plain JSON in an `httpOnly` cookie — `{ email }`. No signing/encryption. Marked as "insecure for prod" in comments.

### Household Registration (`src/features/register-household/`)
- **Purpose:** Creates a `Household` with parent emails and kids after first login.
- **Location:** `src/features/register-household/api/actions.ts`
- **Notes:** Session email + form email are deduped and stored as a JSON array in `emails` column.

### Matching (`src/features/matching/`)
- **Purpose:** Greedy constraint-solver algorithm that assigns households to kids.
- **Strategy:** `CousinExchangeStrategy` — households with most kids matched first; constraint: no household draws its own kids.
- **Interface:** `MatchingStrategy` in `strategies.ts` allows future strategies (Secret Santa, White Elephant).
- **Notes:** Random selection from valid candidates; can deadlock with edge-case distributions (throws on failure).

### Match Reveal (`src/features/match-reveal/`)
- **Purpose:** UI card showing a household's assigned recipients and total gift burden (budget × kids, items × kids).

### Admin Scheduler (`src/features/admin-scheduler/`)
- **Purpose:** Job queue processor. Handles `SEND_OTP` and `SEND_MATCH` job types via Resend email API.
- **Batch size:** 10 jobs at a time, max 3 attempts per job.
- **`SEND_MATCH` job:** Handler stub — not yet implemented.

### Scheduler (`src/shared/lib/scheduler.ts`)
- **Purpose:** Singleton `node-cron` scheduler, started via Next.js instrumentation hook (`src/instrumentation.ts`).
- **Schedule:** Daily at midnight UTC — processes job queue.

### Admin Page (`src/app/admin/page.tsx`)
- **Purpose:** Shows system stats (households, events, pending jobs). Has a button to force-run the scheduler via `GET /api/cron`.
- **Auth:** No authentication guard — admin page is publicly accessible.

## Data Flow

1. User visits `/login`, enters email → `requestOtp` server action creates a `SEND_OTP` job
2. Cron/scheduler fires → `processJobQueue` → Resend sends OTP email
3. User enters OTP → `verifyOtp` validates (currently hardcoded) → sets `session_id` cookie
4. User visits `/dashboard` → if no household, shows `HouseholdRegistrationForm`
5. Household registered → `registerHousehold` creates `Household` + `Kid` records
6. Admin runs matching algorithm → creates `Match` records
7. User visits `/reveal` → fetches `givenMatches` for their household → `RevealCard` shown
